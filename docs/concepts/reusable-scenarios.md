# Reusable Scenarios

A reusable scenario is a named checklist card. It says what inputs it needs,
what acts it runs, and which checked values it can hand back to the caller.

Use a reusable scenario when the same behavior matters in more than one flow:
login, create a user, reserve test data, confirm a message, or check a shared
contract. The point is not to hide behavior. The point is to give repeated
behavior one name and one checked implementation.

Keep four pieces separate while reading examples: inputs, bindings, exports, and
dependencies. Each piece answers a different question.

## Inputs And Bindings

Scenario inputs are blanks on the card. A call fills those blanks with
bindings. Inputs describe what the scenario needs. Bindings provide concrete
values for one call.

In Theater DSL, `scenario messages/make(text: string!)` says `text` is required.
The call `messages/make(text: "hello from Theater")` fills it.

YAML uses the same model with more explicit field names: `inputs.text` declares
the input, and `scenario_calls.bindings.text` passes the value.

## Exports

An export is a value the scenario makes available after a successful act or
successful scenario call. Exported values are useful when the next step must use
something that was discovered or generated by the previous step.

The tutorial example exports `message` inside the library scenario. The flow
then exports that value again as `shared_message`, so the handoff has a name the
next call can read.

## Dependencies

Dependencies are explicit scheduling edges between scenario calls. If one call
uses a value exported by another call, write the dependency too. Theater keeps
the ordering readable instead of inferring it from every `$ref`.

The default dependency predicate is success. YAML writes the same rule as
`when: success`.

## Layout

Repo-aware loading uses two fixed roots:

- `theater/flows/` contains runnable entry flows.
- `theater/lib/` contains reusable scenario library files.

Theater loads a flow from `theater/flows/` and assembles only the library
scenarios that the flow actually calls. It still resolves calls by scenario id,
not by file path. A call to `messages/make` means "find the scenario with this
id", not "import this file".
